Documentation UI

Creating a UI for a technology leader

I designed the documentation UI used by thousands of engineers working with ARM Technologies globally.

Documentation used to come as a hard copy in book format but with a growing need for low reproduction, low maintenance and distribution most product documentation now finds it's home on the web.

Learn how I approached, designed
and tested the solution...

The Challenge

Engineers who build products and technologies need access to a high end user experience that makes browsing and reading ARM technical documentation an easy task.

Must be designed for unknown content (20,000 pages+)

Must render a wide variety of HTML tags including code examples and cater for long line lengths, complex diagrams, and large tabular data.

Support multiple version of the same document

User must be able to change version of a document.

Make technical documentation easy to read

Condense larger hierarchical content into a simple readable format

SEO

Must be able to beat other user generated content such as stack overflow in ranking on a google search.

Support 4 levels of hierarchy

Users must be able to browse and find books, chapters,
topics and sub-topics.

Support a wide range of devices

Must be readable on mobiles and tablets.

Understanding the users

User interviews with engineers were undertaken

I captured:

  • Pain Points
  • Use cases
  • Goals
  • Preferences
  • Common workarounds inherited from the old system

Additional research was undertaken including:
google analytics and competitor analysis.

Key insight 1

Engineering Docs have varying use cases. Including tutorials, product details and topical guides and low-level reference material.

Example document use cases would be:

Reference only - A user looks for reference material on a particular instruction or detail.

Full Chapter - A user learns about a feature by by reading an entire chapter.

Evidence from older online documentation traffic reports from google analytics it was clear the average time on page was low, meaning that most users were skimming or skipping entire pages to find the information they required.

Design Choice: Reduce line length to help users skim content easier.

Key insight 2

Users would typically visit one set of documentation per visit. Meaning that browsing to docs within the same page was not a requirement. Previously all docs had been on the same page.

This changed the user journey to focus more around the product they were related with.

Design Choice: Show one doc per page and direct users to through the parent product page.

Prototyping, user journeys and wireframes

User Journeys

With our key insights and requirements in mind we began to rapidly prototype our UI and design the user journey.

Structuring the sites information architecture so that product documentation belong with it's product page was a typical solution but differed from the past users experience where all docs were held in a singular location.

To ensure users were aware of there path they had taken we added breadcrumbs to the documentation page.

  • Home
  • Products
  • Processors
  • Cortex-A - Cortex-A72
  • Documentation

Wireframes

This began as a series of sketches and Balsamiq / Illustrator mock-ups and was quickly translated into an early HTML prototype and it's level of fidelity was raised until it was ready for user testing.

User Testing

I tested the usability of my prototype with a pool of 10 engineers. This included: software engineers, a hardware engineers and technical authors.

Each user was asked to perform common documentation UI browsing and interaction tasks, their response was captured on video.

The results

Through examination of the results we began to notice a behaviors indicating difficulties in specific interface elements:

  • Locating the toggle version button.
  • Difficulties in browsing and finding 4th level topics.
  • Minor difficulties in recognizing menu hierarchy.
  • Minor difficulties in finding the PDF version of a Document.
  • Minor difficulties in understanding content heading hierarchy.

In total 70% of users were able to complete the test without difficulty.

With the usability issues discovered I made changes to the prototype...

Users had difficulty in finding the right hand version button.

Prototype version 2

Using the usability study results I made changes to the interface to help solve the difficulties users and improve the user experience.

User testing round 2

Subsequent retesting yielded a higher pass rate of usability for all of the usability tests.

I also ran an A/B test to help determine via heat map where users were clicking on each of the prototypes.

The higher heat around the UI on prototype 2 confirmed that more users were interacting with the Menu than the previous prototype. This would not necessarily confirm that the usability was better but that it was encouraging more interaction.

90% of users passed all usability tests on the 2nd round of user testing.
